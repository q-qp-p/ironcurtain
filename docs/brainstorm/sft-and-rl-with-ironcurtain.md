**Open-Weight Models for Autonomous Vulnerability Analysis: Goals, Approach, and Rationale**

**Goal**

Build and openly release a fine-tuned open-weight model capable of supporting the full vulnerability lifecycle: triage of incoming reports, severity assessment, and end-to-end proof-of-concept exploit verification. The target users are individual open source maintainers who currently have no automated tooling, no security team, and no budget for frontier model API access.

**Technical Approach**

The base model is a large open-weight coding model selected for strong performance on agentic software engineering tasks and long-horizon autonomous execution. It is released under a permissive license allowing fine-tuning and local deployment.

Training proceeds in two phases. The supervised fine-tuning phase uses execution traces generated by a finite state machine that models the exploitation process as a sequence of discrete, verifiable states. A human expert guides initial exploration across multiple vulnerability scenarios to establish the correct shape of the state space before decomposition into FSM transitions. This produces synthetic training data that does not exist in any public dataset, avoiding memorization of known exploit patterns.

The reinforcement learning phase uses a separate but structurally related dataset. The reward model is modeled after capture-the-flag competitions: intermediate flags represent required knowledge or actions along the exploitation chain, with partial credit for proximity. This provides dense reward signal across the exploitation reasoning chain rather than sparse terminal feedback, which is the standard failure mode for RL on complex multi-step tasks.

The SFT and RL datasets are deliberately kept distinct to prevent the model from learning the reward structure during the supervised phase.

Training will be validated on smaller models before a full run on a multi-GPU cluster.

**Rationale for Open Release**

The security triage infrastructure for open source software is functionally broken. Bug bounty platforms have largely closed. The CVE and advisory systems require coordination overhead that individual maintainers cannot sustain. In practice, high-severity vulnerabilities are being reported, sometimes patched, and never formally disclosed — leaving downstream users without the information needed to assess their own exposure. Confidential reports are sometimes made public by maintainers before patches exist, removing the reporter's ability to manage the disclosure timeline.

The population most harmed by this failure is also the population least served by existing tooling. Individual open source maintainers cannot afford frontier model API costs, cannot navigate enterprise procurement for commercial security tools, and have no institutional infrastructure for a vetting process. Any access model with friction — API keys, application review, subscription costs — excludes them specifically.

Automated severity assessment directly addresses the bottleneck. A maintainer receiving a disclosure report currently has two options: trust the reporter's severity claim or perform manual analysis requiring expertise they may not have. A locally runnable model that can attempt end-to-end verification and assess exploitability under realistic conditions gives maintainers a third option: independent automated assessment at zero marginal cost.

The dual-use concern is real and is addressed directly rather than elided. The capability is useful to attackers as well as defenders. The argument for open release rests on three observations. First, the target users have no viable alternative access path — gating the model recreates the cost and access barriers that make the current situation broken. Second, the primary capability gap being closed is severity assessment for known reported vulnerabilities, not novel zero-day discovery. Third, the methodology, FSM framework, and training approach are being published simultaneously with the weights, so the research contribution is separable from the weights themselves. Defenders who understand the approach can adapt and extend it; the weights alone are not the ceiling.

The release includes this rationale in full. Responsible release requires naming the tradeoffs, not assuming they will go unnoticed.

This document is the vision; the concrete build has started. The exploitation FSM workflow that generates the memory-corruption traces, and the offline curation pipeline, remain to be built; the capture substrate they depend on does not.
